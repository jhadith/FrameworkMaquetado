MANUAL DE DEFENSA DEL PROYECTO: MARKETPLACE DE TECNOLOGIA

Este documento contiene los argumentos y conceptos tecnicos necesarios para defender el desarrollo de la aplicacion de pies a cabeza frente a evaluadores y profesores.



1. INTRODUCCION Y RESUMEN GENERAL

El proyecto es una plataforma web de comercio electronico (Marketplace) para productos tecnologicos.
El desarrollo esta construido con HTML5, JavaScript nativo (Vanilla JS) y Bootstrap 5 como framework de diseño.
El sistema opera completamente del lado del cliente (Client-Side), permitiendo simular un flujo completo de seleccion, persistencia, validacion y compra sin depender de servidores externos.



2. ARQUITECTURA Y FLUJO DE DATOS DEL SISTEMA

Paso A: Carga del Catalogo (tienda.js)
El catalogo se inicia leyendo un arreglo local de productos (base de datos en memoria).
Los datos se inyectan en el elemento #productos-grid mediante JavaScript manipulando el DOM.

Paso B: Agregacion de Productos al Carrito
Al pulsar Añadir, se verifica si el item ya existe en el carrito.
Si existe, se incrementa su cantidad. Si no, se añade el objeto con cantidad inicial 1.

Paso C: Persistencia en Almacenamiento Local (LocalStorage)
Cada cambio en el carrito llama a guardarYActualizarCarrito().
Se convierte el arreglo de objetos a cadena de texto con JSON.stringify().
Se almacena bajo la clave 'carrito' en el navegador, permitiendo persistencia al recargar la pagina o navegar a otra.

Paso D: Recuperacion de Datos en la Factura (facturacion.js)
Al cargar factura.html, se recupera el string JSON de localStorage.
Se revierte a objeto JavaScript con JSON.parse() para dibujar el resumen en pantalla.

Paso E: Formulario y Validaciones Nativas
El usuario ingresa informacion de envio y pago.
Atributos como required y type="email" ejecutan validaciones en el navegador antes de procesar los datos.

Paso F: Validacion Matematica de la Tarjeta (Algoritmo de Luhn)
comprobarNumeroTarjeta() ejecuta la formula modulo 10 (Luhn) sobre los digitos limpios de la tarjeta.
Duplica digitos alternos y acumula los resultados para comprobar si la suma total es divisible para 10.
Garantiza que el numero ingresado tenga la estructura de una tarjeta real antes de procesar el pago.

Paso G: Generacion y Serializacion de Datos (XML y JSON)
Una vez validada la tarjeta, enviarFactura() crea dos estructuras de datos:
Estructura JSON: Formato estandar moderno e interoperable, optimo para APIs REST y bases de datos NoSQL.
Estructura XML: Formato jerarquico tradicional con secciones CDATA, ideal para sistemas de facturacion electronica y compatibilidad gubernamental.
Ambos formatos se imprimen en la consola web para certificar el modelamiento de datos.

Paso H: Cierre y Limpieza del Carrito
Al pulsar Volver a la Tienda en el recibo final, se remueve la clave 'carrito' de localStorage para reiniciar el flujo.



3. DEFENSAS CLAVE Y JUSTIFICACION DE TECNOLOGIAS

¿Por que JavaScript nativo (Vanilla JS)?
Elimina la necesidad de dependencias externas pesadas o compiladores.
El codigo se ejecuta de inmediato en cualquier navegador web moderno.
Provee un control total y directo sobre los elementos del DOM y el almacenamiento.

¿Por que LocalStorage?
Ofrece almacenamiento persistente de clave-valor directamente en el cliente (hasta 5MB).
Evita tener que configurar un backend para mantener el estado de la compra del usuario entre paginas.

¿Por que usar JSON y XML en conjunto?
Demuestra la versatilidad de la aplicacion para comunicarse con diferentes tipos de backend.
JSON representa la modernidad y la ligereza. XML representa la robustez y el cumplimiento normativo.

¿Que funcion cumplen etiquetas como <i> y <svg> y que pasa si se eliminan?
La etiqueta <i> se asocia con Lucide Icons para renderizar graficos vectoriales definidos por clases CSS.
La etiqueta <svg> incrusta de manera nativa graficos basados en vectores (escalables sin perder calidad).
El elemento <symbol> agrupa trazados vectoriales, y <use> los clona en cualquier parte del HTML por su ID.
Si se eliminan estas etiquetas, los iconos visuales (carrito, estrellas, check de exito) desapareceran por completo de la pantalla, dejando la interfaz huerfana de elementos visuales basicos y dañando la UX del usuario.



4. DEFENSA CENTRAL DE BOOTSTRAP 5

Justificacion del Uso de Bootstrap
Acelera el proceso de maquetacion al proveer componentes listos para usar y pre-estilizados.
Garantiza consistencia visual a lo largo de todas las pantallas del aplicativo.
Es responsive de forma nativa, adaptandose sin escribir CSS personalizado extenso.
Ofrece accesibilidad y compatibilidad nativa entre navegadores mediante variables CSS y normalizacion.

El Sistema de Grilla Responsiva (Grid System)
Se fundamenta en un diseño de 12 columnas flexibles (Flexbox).
La clase row define el contenedor de la fila.
La clase col indica distribucion automatica de espacio.
La clase col-md-5 col-lg-4 order-md-last en factura.html se adapta asi:
En dispositivos moviles pequeños (menor a 768px), el resumen del carrito ocupa el 100% de la pantalla y se posiciona abajo/arriba segun el flujo HTML normal.
En pantallas medianas (md: tablets, >=768px), ocupa 5 de las 12 columnas.
En pantallas grandes (lg: escritorios, >=992px), ocupa 4 de las 12 columnas.
La propiedad order-md-last fuerza que en tablet y escritorio se ordene al final (derecha), manteniendo una disposicion de lectura natural.
La clase row-cols-1 row-cols-sm-2 row-cols-md-3 g-4 en index.html adapta el catalogo a 1 columna en movil, 2 en tablet y 3 en escritorio de forma automatica.

Componentes Clave Implementados y su Adaptacion
Offcanvas (Panel lateral #cart-sidebar): Crea un menu deslizable interactivo desde el borde derecho (offcanvas-end). Es ideal para carritos de compras porque no interrumpe la navegacion del usuario en la pantalla principal.
Modal (Ventanas emergentes #product-details-modal y #facturaModal): Capas flotantes sobrepuestas al contenido. Se usan para visualizar informacion extendida y confirmar acciones sin redirigir al usuario.
Cards (Tarjetas de productos): Estructura modular con bordes, padding y sombras para organizar la imagen, el precio y el boton de compra de cada item.
Badges (Etiquetas numericas #cart-count): Pequeñas capsulas de color (bg-danger, bg-primary) usadas para indicar el numero de items en tiempo real.
Dropdowns (Selector de tema #bd-theme): Despliega opciones flotantes para modificar configuraciones especificas en pantalla de manera compacta.

Utilidades CSS de Bootstrap Usadas
d-flex: Activa flexbox de manera inmediata para alinear cajas de datos.
align-items-center y justify-content-between: Centran y espacian de forma dinamica elementos hijos de contenedores flexibles.
gap-*: Aplica espaciado uniforme entre elementos hijos dentro de un contenedor Flexbox.
shadow-sm: Añade una sombra ligera para dar profundidad y estetica de relieve moderno.
text-truncate: Trunca textos largos que podrian desbordar la tarjeta, añadiendo puntos suspensivos automáticamente.
px-*, py-*, m-*, mt-*, mb-*: Manejan paddings (rellenos) y margins (margenes) en ejes especificos sin necesidad de CSS personalizado.

Soporte de Tema Claro y Oscuro (Data Attributes)
Bootstrap 5.3+ admite modo oscuro nativo mediante el atributo data-bs-theme.
La aplicacion define data-bs-theme="auto" para leer la preferencia del sistema operativo de manera predeterminada.
El script de inicializacion lee localStorage para aplicar el tema de forma persistente.
La funcion setTheme() manipula el atributo del elemento raiz de HTML (document.documentElement) para aplicar las variables de color correspondientes de Bootstrap de forma automatica en toda la aplicacion.
